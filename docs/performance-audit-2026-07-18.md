# 工时系统并发审计与压测报告（2026-07-18）

## 结论

在本机 WSL（8 vCPU、约 5.8 GiB 内存，PostgreSQL/Redis/Node 同机）上，系统足以承载约 1000 人周五集中提交。

- 原始四实例提交峰值约 **464.91 req/s**，并发继续升到 300 后吞吐回落。
- 增加鉴权缓存后，四实例激进配置峰值达到 **891.77 req/s**，提高约 **91.8%**。
- 推荐的安全部署（4 个 API、每实例池 12、API 只入队、独立审批 worker）在 100 并发时为 **537.66 req/s**；1000 次多人同周提交为 **585.37 req/s**，成功率均为 100%。
- Redis 峰值 CPU 约 0.2%，不是瓶颈；高压时主要压力在 PostgreSQL，其次是 Node。
- 入口合理并发区间约 100–300；超过 300 后延迟和数据库争用明显增加，应通过网关排队/背压，而不是继续放大连接数。

## 页面初始化读峰值

新增共享参考数据缓存后，以 100 并发模拟 1000 次页面打开：

- 进行中项目 + 系统设置：2000/2000 成功，**1783.66 HTTP req/s**，页面 p95 366.4 ms。
- 项目 + 设置 + 部门 + 组织树：4000/4000 成功；热缓存 **1637.27 HTTP req/s**，冷缓存 **1491.83 HTTP req/s**。

缓存策略：

- JWT 鉴权快照：60 秒，登录时预热；改密、登出、用户/角色变更主动失效。
- 组织目录：60 秒；部门、分组、用户、角色写操作主动失效。
- 进行中项目、项目审批上下文、默认审批流、系统设置：300 秒；对应写操作主动失效。
- 单实例内合并同一 key 的并发冷加载，避免缓存过期瞬间形成回源风暴。

## 已实施的并发正确性改进

1. 审批后置由 Redis List 改为 Redis Streams consumer group。
2. 消息仅在业务处理成功后 ACK；worker 崩溃后，超过 30 秒未确认的消息由健康实例接管。
3. `approval_instances(targetType, targetId)` 增加唯一索引，保证至少一次投递不会创建重复审批实例。
4. worker 重试前先复用已存在实例，覆盖“实例已落库但消息尚未 ACK”的崩溃窗口。
5. 恢复演练结果：模拟宕机后 pending=1；自动接管后 pending=0，审批实例数量仍为 1。
6. 全局限流在登录后按令牌哈希隔离，未登录才按 IP；登录限流只累计失败请求，避免公司 NAT 出口下 1000 人共享同一额度。

## 推荐生产拓扑

- 4 个 API 实例：`APPROVAL_WORKER=0`、`DB_POOL_MAX=12`。
- 1 个独立审批 worker：`npm run worker:approval:prod`，进程级覆盖 `DB_POOL_MAX=8`；有持续积压时再增加到 2 个。
- PostgreSQL 总连接预算约 56，加上运维/迁移连接仍低于常见 100 连接上限。
- 网关建议将提交接口的在途并发控制在 200–300，超出时短队列等待，避免把压力直接放大到数据库。
- 应用与 PostgreSQL 分机后再做一次同脚本复测；当前资源采样表明分机比扩 Redis 更有收益。

## 验证结果与注意事项

- 服务端 TypeScript 编译通过。
- Vitest：6 个文件、68 项测试全部通过。
- migration 已执行，并完成一次 `revert → run` 可逆验证。
- 最终数据检查：审批缺失 0、重复实例 0、Stream pending 0、lag 0。
- 压测在测试库写入了大量未来日期、描述以“压测提交”或“同周并发提交”开头的数据；未自动删除，避免未经授权做破坏性数据操作。
- 对比测试在同一机器完成，但数据库随压测逐步增大，因此不是严格的隔离实验；上线前应使用脱敏生产数据快照、固定数据库快照和独立压测机再验收一次。
- 日志还显示既有 AI worker 在 production 构建下加载 `pi-coding-agent` 时出现 exports 错误；不影响工时主链路，但会影响 AI 助手，建议另行修复。

## Caddy、3/4 API 与 worker 批量复测

在 4 vCPU、7.8 GiB WSL 中改用 Caddy `least_conn`，API 仅入队，独立审批 worker 的 PostgreSQL 连接池降为 6。每组测试均执行以下固定流程：恢复同一个 PostgreSQL custom-format 快照、清理 `worktime:*` Redis 命名空间、执行 `ANALYZE`、使用固定空白日期区间预热，再运行正式测试。快照 SHA-256 为 `d27838a6b120733b50490fdab6a3e8b3606bb61053fcc909c5c5b8a73dbebd0e`。

最初固定的 2973–3030 年区间与快照中 3601 条历史压测记录重叠，会触发审批版本化更新，不能与“新增提交”基线比较；最终使用快照中没有记录的 9489–9738 年区间。重叠数据的结果只作为诊断，不计入下表。

### 最终建议

- 4 核环境使用 **Caddy + 3 API + API 每实例 DB_POOL_MAX=12 + 独立 worker DB_POOL_MAX=6 + APPROVAL_BATCH_SIZE=50**。
- 网关正常在途并发设为 **200**；250 以上进入争用区，不作为安全容量。
- 当前稳定容量约 **612–645 req/s**，p95 约 **497–575 ms**。在应用、worker 和 PostgreSQL 同机的前提下，尚未达到 700–800 req/s；继续增加第 4 个 API 不能解决瓶颈。
- 下一阶段若必须达到 700–800 req/s，应优先将审批 worker 或 PostgreSQL 与 API 隔离 CPU，再复测；其次才是查询计划和事务级热点分析。
- 成功请求日志在生产默认采样 1%，4xx/5xx 始终记录。实测 600 个成功健康请求记录 5 条采样日志，另一个 401 请求完整记录。
