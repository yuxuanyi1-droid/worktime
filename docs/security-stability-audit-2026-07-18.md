# 安全、稳定性与 2000 人容量审计（2026-07-18）

## 结论

本轮未改变工时、审批、报表等基本业务语义，已修复确认的高风险凭据问题、多实例登录防护缺口、OIDC 回调保护失效、审批队列半提交风险和管理员待办全量加载问题。服务端与客户端生产依赖审计均为 0 个已知漏洞；新增迁移均完成执行、回滚、再次执行验证。

当前 4 核 8 GB 推荐配置保持：Caddy + 3 API（每实例 `DB_POOL_MAX=16`）+ 独立审批 worker（`DB_POOL_MAX=6`、批量 50）+ PostgreSQL + Redis。该配置适合 1000–2000 人公司日常错峰访问，但不能解释为支持 2000 人同时提交。建议继续以约 200 个在途请求作为单机延迟安全区间。

## 已修复问题

| 级别 | 问题 | 处理结果 |
|---|---|---|
| 高 | PAT 明文长期存库、登录自动创建 PAT，数据库泄露即可直接调用 API | PAT 改为仅创建时展示一次，数据库只保留 SHA-256 与脱敏前缀；历史明文已清空；内部 AI 改用 2 小时短期 JWT |
| 高 | 多实例登录失败次数只存在各进程内存，可轮询实例绕过锁定 | Redis 共享失败计数；Redis 故障时回退本实例内存；用户名与 IP 组合键经哈希处理 |
| 高 | OIDC 回调限流挂载路径没有匹配 `/:provider/callback` | 限流器移动到真实回调路由；回调基地址增加同源/允许列表校验 |
| 高 | 审批实例、任务和工时回写不在同一事务，进程中断可能留下半成品 | 每批审批处理改为数据库原子事务；通知仍在提交后发送；Redis Streams 保持至少一次投递和幂等约束 |
| 中 | 管理员待办先加载公司全部记录再内存分页 | 改为数据库过滤、排序和分页；新增 `(status, updatedAt)` 索引 |
| 中 | 管理员随机重置密码和 JIT 隐藏密码使用 `Math.random` | 改用 `crypto.randomBytes`；新密码下限 8 位；不存在用户也执行 bcrypt，降低用户名枚举风险 |
| 中 | API 无安全响应头、暴露 Express 标识，生产指标可能匿名访问 | 启用 Helmet/CSP，关闭 `x-powered-by`；生产未配 `METRICS_TOKEN` 时指标端点隐藏 |
| 中 | 生产 API 默认监听全部网卡，可能绕过 Caddy 直连 | 生产默认监听 `127.0.0.1`，容器部署可显式设置 `API_HOST=0.0.0.0` |
| 中 | PAT 每次请求查询两次用户并写一次使用时间 | 合并为一次关系查询，使用时间最多每 5 分钟更新一次 |
| 中 | 多 API 启动时可同时执行 migration；表枚举触发 pg 未来弃用警告 | PostgreSQL advisory lock 串行化 schema/migration 阶段，并改用轻量表计数 SQL |
| 中 | API 收到 SIGTERM 直接退出 | 增加最长 30 秒优雅下线，停止接流量后再关闭 worker、Redis 和数据库 |
| 中 | ECharts 5.6.0 存在已公开 XSS 漏洞 | 升级到 6.1.0，客户端生产构建通过 |

## 1000–2000 人容量判断

“并发 200”是 200 个同时在途的 HTTP 请求，不等于 200 名员工。真实员工会阅读页面、思考和间歇操作；因此 2000 名员工日常使用通常远低于 2000 个同时在途请求。按当前最重的“连续提交工时”模型，单机每秒可完成约 575 次提交，远高于一般公司的真实业务到达速率。

以下情况需要扩容：全员被要求在同一分钟集中提交、报表大范围导出与提交同时发生、AI/附件等长请求大量并发，或长期监控出现 p95 超过 600 ms、数据库连接等待、审批队列持续增长。

## 破千请求每秒的优先路线

1. 将 PostgreSQL（优先）和审批 worker 移到独立服务器，避免与 API 争抢 4 核 CPU、内存带宽和磁盘；Redis 可同数据库服务器起步，但生产建议独立或使用托管高可用实例。
2. API 继续保持每实例连接池 16；实例数增加时在 PostgreSQL 前增加 PgBouncer transaction pooling，控制数据库物理连接总数。
3. 扩到 2 台应用服务器、每台 3 个 API，由外层负载均衡分流。按当前保守值不能简单线性翻倍，目标应先验收 900–1100 req/s、p95<600 ms。
4. 将大范围报表导出改为异步任务；高频统计使用预聚合表/定时快照；必要时为报表增加只读副本，避免与提交写入争用。
5. 用 Prometheus 持续观察 API p95、PostgreSQL 活跃连接/锁等待、Redis Streams lag、错误率和事件循环延迟，依据指标扩容，而不是依据员工总数扩容。

## 仍需在上线前完成

- 生产启用 HTTPS，并设置高强度 `JWT_SECRET`、`METRICS_TOKEN`、数据库/Redis 密码；密钥进入专用 secret 管理，不放代码库。
- PAT 的 `scopes` 当前只是预留字段，外部自动化大量使用前应实现最小权限 scope，并为 PAT 单独设置限流。
- 为登录、改密、PAT 创建/撤销、OIDC 绑定和越权访问增加集成测试；在 CI 中固定执行测试、构建、依赖审计和 migration dry-run。
- PostgreSQL 建立定期备份和恢复演练；Redis Streams 配置队列积压告警；Caddy/应用服务器做进程守护和开机启动。
- 对报表日期范围和导出规模增加产品级限制或异步导出策略，避免合法账号发起超大查询拖慢在线请求。
